# AI駆動開発ガイド：チームでの活用（非エンジニア向け）

このガイドは、AI駆動開発をチームでどう活用するか、PM・デザイナー・経営者の視点から解説します。

> **エンジニアの方へ**: 技術的な詳細は [実践編（エンジニア向け）](./ai_development_practice.md) を参照してください。

> **はじめての方へ**: 先に [AI駆動開発ガイド：はじめに](./ai_development_setup_nontechnical.md) をお読みください。

## このガイドで学べること

- AI駆動開発の全体フロー
- 各フェーズで非エンジニアができること
- チーム運用のベストプラクティス

## AI駆動開発の全体フロー

```
機能起票 → 設計相談 → 設計決定の記録 → タスク分解
                                          ↓
ドキュメント更新 ← レビュー ← 実装 ← 仕様確認
```

### 各フェーズの概要

| フェーズ | やること | 関わる人 |
|---------|---------|---------|
| 機能起票 | 何を作るか決める | PM、ステークホルダー |
| 設計相談 | どう作るか議論 | エンジニア（AIと壁打ち） |
| 設計決定の記録 | 決定事項を文書化 | エンジニア |
| タスク分解 | 作業を細かく分ける | エンジニア、PM |
| 実装 | コードを書く | エンジニア（AIが支援） |
| レビュー | コードをチェック | エンジニア（AIが一次チェック） |
| ドキュメント更新 | 説明書を更新 | エンジニア（AIが生成支援） |

## CLAUDE.md：チームの「共有脳」

### CLAUDE.mdとは？

プロジェクトのルールや情報をまとめたファイルです。AIが自動的に読み込み、プロジェクト固有の文脈を理解した回答をしてくれます。

### なぜ重要か？

**Before（CLAUDE.mdなし）**:
- 「うちのプロジェクトはTypeScriptを使っています」
- 「DBはPostgreSQLです」
- 「コーディング規約は...」
- → 毎回AIに説明が必要

**After（CLAUDE.mdあり）**:
- AIが事前にプロジェクト情報を把握
- 説明なしで適切な回答
- チーム全員が同じ前提で作業

### 非エンジニアが追加できる情報

エンジニアに以下の情報をCLAUDE.mdに追加してもらうよう依頼できます：

```markdown
# ビジネス情報（PMが提供）

## プロダクトの目的
〇〇業界向けのB2Bサービス。中小企業の業務効率化を支援する。

## 主要なユーザーペルソナ
- 経理担当者（30-50代、ITに詳しくない）
- 経営者（時間がない、モバイルで確認したい）

## 優先すべき価値
1. 使いやすさ > 機能の豊富さ
2. 安定性 > 新機能のスピード
3. セキュリティは最優先

## ビジネス上の制約
- 月末はアクセスが集中する
- 3月決算期はダウンタイム厳禁
```

## 設計フェーズでの活用

### 設計壁打ちとは？

エンジニアがAIに「この機能をどう実装すべき？」と相談し、選択肢と根拠を整理することです。

### 非エンジニアとして提供すべき情報

エンジニアが設計壁打ちをする際、以下の情報があると精度が上がります：

| 情報 | 例 |
|------|-----|
| ビジネス要件 | 「ユーザーは1日100件のファイルをアップロードする想定」 |
| 優先順位 | 「スピード重視。多少の制限は許容」 |
| 制約条件 | 「予算は月10万円以内」 |
| 将来の展望 | 「半年後にはモバイルアプリも対応予定」 |

### ADR（アーキテクチャ決定記録）

設計の意思決定を記録したドキュメントです。

**PMが知っておくべきこと**:
- なぜその技術を選んだかが書いてある
- 将来「なぜこうなってるの？」と思ったら読む場所
- `/docs/adr/` ディレクトリに保存される

**ADRの例（要約）**:
```
問題: ファイルを非同期で処理する仕組みが必要

決定: Amazon SQS + ワーカーを採用

理由:
- 既存インフラとの相性が良い
- 拡張が容易
- 運用コストが予測しやすい

却下した案:
- Redis: 別途インフラ費用がかかる
- DBポーリング: 負荷が高くなる
```

## 実装フェーズでの活用

### エンジニアへの要件の伝え方

AIを活用する開発では、要件の明確さが成果物の品質に直結します。

**曖昧な依頼（AI精度が下がる）**:
```
「いい感じのログイン画面を作って」
```

**具体的な依頼（AI精度が上がる）**:
```
「ログイン画面を作ってほしい。
- メールアドレスとパスワードで認証
- パスワードを忘れた場合のリンクも必要
- モバイルでも使いやすいレイアウト
- 既存のデザインシステムに合わせる」
```

### 進捗確認のポイント

AI駆動開発では、従来と進捗の見え方が変わることがあります。

| 従来 | AI駆動開発 |
|------|-----------|
| 「〇〇ファイルを作成中」 | 「AIに〇〇を生成させて、レビュー中」 |
| 「調査に時間がかかっている」 | 「AIと設計を壁打ち中」 |
| 「コードレビュー待ち」 | 「AIレビュー済み、人間レビュー待ち」 |

## 品質フェーズでの活用

### AIによるコードレビュー

PR（プルリクエスト）を出す前に、エンジニアがAIにレビューさせます。

**AIがチェックすること**:
- セキュリティの問題（認証漏れ、データ漏洩リスク）
- パフォーマンスの問題（遅くなりそうな処理）
- コーディング規約への準拠
- エラー処理の漏れ

**メリット**:
- 人間のレビュー時間が短縮
- 見落としが減る
- 一貫した基準でチェック

### 非エンジニアが確認できること

コードの詳細は分からなくても、以下は確認できます：

- **PRの説明文**: 何を変更したか、なぜ変更したかが書いてある
- **テスト結果**: 緑色なら成功、赤色なら失敗
- **AIレビューの結果**: 指摘があったかどうか

## ドキュメント自動生成

### 何が自動生成できるか

| ドキュメント | 説明 |
|-------------|------|
| README | プロジェクトの概要・セットアップ手順 |
| API仕様書 | 外部と連携するためのインターフェース説明 |
| コミットメッセージ | 変更内容の要約 |
| PR説明文 | プルリクエストの概要 |
| 変更履歴 | バージョンごとの変更点 |

### PMとして活用できる場面

- **進捗報告**: 「今週のコミット内容をまとめて」とAIに依頼
- **リリースノート**: 変更履歴からユーザー向け説明を生成
- **技術調査**: 「このコードは何をしている？」とAIに質問

## チーム運用のベストプラクティス

### 1. CLAUDE.mdを育てる

最初から完璧を目指さず、少しずつ情報を追加していきます。

**Week 1**: 基本的な技術スタックを記載
**Week 2**: コーディング規約を追加
**Week 3**: よく使うコマンドを追加
**Week 4**: ビジネス上の制約を追加

### 2. PR前AIレビューをルール化

チームのルールとして「PRを出す前にAIレビューを実施」を設定します。

```markdown
## PRを出す前のチェックリスト

- [ ] テストが通ること
- [ ] AIレビューを実施し、指摘事項を修正済み
- [ ] PR説明文を記載
```

### 3. 設計決定を必ず記録

「口頭で決めた」で終わらせず、ADRとして記録することを習慣化します。

### 4. 振り返りにAIを活用

スプリント振り返りで：
- 「このスプリントの変更をまとめて」
- 「技術的負債として残っている箇所は？」
- 「次に改善すべき点は？」

## よくある失敗パターンと対処法

### 1. AIに丸投げ

**症状**: 「全部AIにやらせればいい」と考え、要件も曖昧なまま依頼

**対処法**:
- 要件は人間が明確に定義する
- AIはあくまで「実行」を支援するツール
- 判断と意思決定は人間が行う

### 2. AIの出力を盲信

**症状**: AIが書いたコードをレビューなしで本番投入

**対処法**:
- 人間によるレビューを必須に
- テストを必ず実行
- セキュリティチェックを怠らない

### 3. コミュニケーション不足

**症状**: 「AIがあるから聞かなくていい」と思い込む

**対処法**:
- AIは補助であり、チーム内コミュニケーションは引き続き重要
- 設計判断は人間同士で議論
- AIの提案をチームで検討

### 4. 期待値のずれ

**症状**: 「AIがあれば開発速度が3倍になる」と期待しすぎ

**対処法**:
- 最初は学習コストがかかる
- 効果が出るまで1-2ヶ月かかる
- 段階的な導入を計画

## 成功指標

### 定量的指標

| 指標 | 測定方法 |
|------|---------|
| PRリードタイム | 作成から承認までの時間 |
| バグ発生率 | リリース後のバグ件数 |
| ドキュメントカバレッジ | ドキュメント化されている機能の割合 |
| 設計相談の時間 | 設計決定までにかかる時間 |

### 定性的指標

- エンジニアの満足度
- 新人のオンボーディング速度
- チーム内のコミュニケーション品質
- 技術的負債の可視化度合い

## コスト管理

### 費用の構成

| 項目 | 費用目安 |
|------|---------|
| ツール利用料 | 月額数千円〜数万円/人 |
| 学習コスト | 初月は生産性が一時的に低下する可能性 |
| 運用整備 | ルール策定・ドキュメント整備の工数 |

### 効果との比較

| 削減できるコスト |
|-----------------|
| 技術調査の時間 |
| ドキュメント作成の時間 |
| バグ修正の時間 |
| 新人教育の時間 |
| コードレビューの時間 |

### ROIの考え方

- 短期（1-2ヶ月）: 学習コストで投資フェーズ
- 中期（3-6ヶ月）: 効率化効果が現れ始める
- 長期（6ヶ月〜）: 継続的な効率化とコスト削減

## まとめ：非エンジニアとしてできること

### PMとして

- 要件を明確に言語化して伝える
- CLAUDE.mdにビジネス情報を追加してもらう
- AIの提案を鵜呑みにせず、ビジネス視点でレビュー
- 進捗報告にAIを活用

### デザイナーとして

- デザイン要件を具体的に伝える
- デザインシステムの情報をドキュメント化
- 実装可能性の確認にAIを活用
- デザインレビューにAIの視点を取り入れる

### 経営者・マネージャーとして

- 導入の意思決定を行う
- 適切なプロジェクトを選定
- 効果測定の基準を設定
- セキュリティポリシーを策定
