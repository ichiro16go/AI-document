## 全体フロー

```
機能起票 → 設計壁打ち → ADR作成 → タスク分解
                                        ↓
ドキュメント生成 ← AIレビュー ← 実装（Claude Code）
```

すべてのフェーズに共通する基盤が **CLAUDE.md**。ここにプロジェクトの文脈を書くことで、チーム全員とAIが同じ前提で動ける。

## セットアップ：CLAUDE.md

リポジトリのルートに置くと、Claude Codeが自動的に読み込む。チームでGit管理する。

```markdown
# プロダクト概要
〇〇 SaaS。ユーザーが〇〇をするB2Bサービス。

# チーム体制・技術スタック
- フロント: Next.js 15 (App Router) + TypeScript
- バックエンド: NestJS + PostgreSQL + Prisma
- モバイル: React Native (Expo)
- インフラ: AWS ECS + RDS, Terraform管理

# アーキテクチャ判断の原則
- DBスキーマ変更は必ずmigrationファイルで管理
- APIはOpenAPI specを先に書いてから実装する
- モバイルとWebでAPIは共通化する

# よく使うコマンド
- `pnpm dev`          開発サーバー起動
- `pnpm test:watch`   テスト監視
- `pnpm lint`         Lintチェック

# 禁止事項
- any型禁止
- raw SQL禁止（Prisma ORMを使うこと）
- secretsをコードにハードコード禁止
- console.logをコミットしない
```

**ポイント：** 「毎回AIに説明する手間」がなくなる。新メンバーへの文脈共有も自動化される。

## 設計フェーズ

### 1. 設計壁打ち

「実装して」の前に、必ずこのフォーマットでClaudeと相談する。

```
## 背景
ユーザーがファイルをアップロードして、非同期で変換処理をする機能を追加したい。

## 制約
- 既存スタックはNestJS + PostgreSQL + AWS ECS
- 1ファイル最大100MB
- 変換に最大10分かかる場合がある

## 相談したいこと
1. キューイングの方式（SQS vs Bull vs DB polling）の選択基準
2. フロントエンドへの進捗通知の方法（Polling vs WebSocket vs SSE）
3. 失敗時のリトライ戦略

懸念点や見落としているトレードオフがあれば教えてほしい。
```

Claudeは「SQSを選ぶ場合のIaCコスト」「SSEとWebSocketのモバイル対応の違い」など、自分では気づかない観点を返してくれる。

### 2. ADR（アーキテクチャ決定記録）の作成

設計相談の結果を流さないために、毎回ADRとして記録する。

```
さっきの会話を元に、ADRファイルを
`/docs/adr/003-async-file-processing.md` として書いて。
フォーマットは「状況・決定・根拠・却下した選択肢」で。
```

ADRのテンプレート：

```markdown
# ADR-003: 非同期ファイル処理のアーキテクチャ

## 状況
ユーザーがアップロードしたファイルを非同期で変換する機能が必要。

## 決定
Amazon SQS + ECSワーカーの構成を採用する。

## 根拠
- 既存インフラ（AWS）との親和性が高い
- スケールアウトが容易
- ジョブの可視性・デッドレターキューが標準機能

## 却下した選択肢
- Bull (Redis): 別途Redisインフラが必要でコスト増
- DB polling: スケールしない、DBに負荷がかかる
```

### 3. フロント・バック間のインターフェース先行定義

実装前にAPI仕様だけ書かせると、並行開発中のすれ違いを防げる。

```
フロントエンドとバックエンドが並行して開発できるよう、
ファイルアップロード機能のOpenAPI specを先に書いて。
実装はしなくていい。
```

## 実装フェーズ

### Claude Codeの基本的な使い方

**悪い例：**
```
認証を実装して
```

**良い例：**
```
JWTを使ったログイン機能を実装して。
- 配置先: /src/auth/
- テスト: Vitestで書くこと
- 既存の User モデル（/src/models/user.ts）を使うこと
- パスワードはbcryptでハッシュ化
```

具体的なスコープ・場所・制約を渡すほど品質が上がる。

### Planモード（大きなタスクの前に）

大きなタスクを依頼する前に、まず計画を立てさせることで方向性のズレを防げます。

```bash
# Claude Codeを起動
claude

# 対話中に計画を依頼
> ユーザー認証システム全体を設計・実装したい。まず計画を立てて。
```

または、明示的にPlanモードを指示:

```bash
# 対話中にPlanモードを使用
> /plan ユーザー認証システムの実装
```

計画が出力されたら、方向性を確認してから「この計画で進めて」と指示します。大きなリファクタリングや新機能追加の前に特に有効。

### MCP（Model Context Protocol）でツールを拡張する

MCPを使うと、Claude Codeに外部ツールとの連携機能を追加できます。

```bash
# MCPサーバーの追加（設定ファイルに保存される）
claude mcp add <サーバー名> -- <コマンド>

# 例: ファイルシステムサーバーの追加
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem ~/dev

# 設定済みのMCPサーバー一覧を確認
claude mcp list
```

**利用可能なMCPサーバー例**:
- `@modelcontextprotocol/server-filesystem`: ファイルシステム操作
- `@modelcontextprotocol/server-github`: GitHub連携
- `@modelcontextprotocol/server-postgres`: PostgreSQL接続

MCPの詳細は[公式ドキュメント](https://modelcontextprotocol.io/)を参照してください。

これにより「現在のDBスキーマを見て、N+1クエリを見つけて直して」といった指示が現実的になる。

### チームで並列実行する

```bash
# フロントエンド担当
claude "アップロードUIを実装して。/src/features/upload/ に配置。
進捗表示はSSEで /api/jobs/:id/status をポーリングすること"

# バックエンド担当（別ターミナル）
claude "ファイルアップロードAPIとジョブキュー処理を実装して。
SQSは既存の /infra/sqs.tf を参考に追加すること"
```

CLAUDE.mdに技術スタックと制約が書いてあるので、どちらも余計な説明なしに正しい方向に動く。

### テストを先に書かせる

```
実装はしなくていい。
ファイルアップロード機能のテストコードだけVitestで書いて。
- 正常系: アップロード成功→ジョブIDが返る
- 異常系: ファイルサイズ超過、未認証アクセス
```

テスト仕様を固めてから実装させると、動作の定義が明確になる。

## 品質フェーズ

### PR前のAIセルフレビュー（チームルールに追加）

各メンバーがPRを出す前に実行するルールにする。

```bash
git diff main | claude "このPRをレビューして。
特に以下の観点でコメントして：
1. セキュリティ上の問題（認証・認可・インジェクション）
2. CLAUDE.mdの規約違反
3. N+1クエリなどのパフォーマンス問題
4. エラーハンドリングの漏れ"
```

### デバッグ支援

「動かない」ではなく、スタックトレース・再現手順・期待する動作を丸ごと貼り付けて送る。

```
以下のエラーが出て動かない。直してほしい。

【エラー】
TypeError: Cannot read properties of undefined (reading 'id')
  at UserService.getProfile (/src/user/user.service.ts:45)

【再現手順】
1. /api/users/profile にGETリクエスト
2. Authorizationヘッダーなしで送信

【期待する動作】
401 Unauthorized を返す

【実際の動作】
500 Internal Server Error
```

## よくある失敗パターンと対処法

### 1. AIが古い情報を提案してくる

**症状**: 非推奨のAPIやライブラリの古いバージョンの書き方を提案される

**対処法**:
```
Next.js 15のApp Routerを使っている。
pages/ディレクトリは使用していない。
最新のドキュメントに基づいて実装して。
```
- バージョンを明示的に伝える
- 公式ドキュメントのURLを一緒に渡す

### 2. AIが無限ループに陥る

**症状**: 同じエラーを修正→別のエラー発生→元に戻すを繰り返す

**対処法**:
```
一度立ち止まって。現在の状況を整理して。
1. 何を達成しようとしているか
2. 何が問題になっているか
3. 試した解決策とその結果
を箇条書きで教えて。
```
- 「立ち止まって」「整理して」と明示的に指示
- 必要に応じて新しいセッションを開始

### 3. 変更範囲が意図より大きい

**症状**: 1つの機能を頼んだのに、関連ファイルを大量に変更される

**対処法**:
```
変更範囲を最小限にして。
触っていいファイルは /src/features/upload/ のみ。
他のファイルは変更しないで。
```
- 変更していいファイル/ディレクトリを明示
- 「リファクタリングは不要」と伝える

### 4. コードが動かない状態でコミットされる

**症状**: AIがコミットを提案するが、テストが通っていない

**対処法**:
- コミット前に必ず動作確認
- CLAUDE.mdに「コミット前にテストを実行すること」と記載
```
# CLAUDE.mdに追記
## コミット前のチェックリスト
- [ ] pnpm test が通ること
- [ ] pnpm lint が通ること
- [ ] 動作確認が完了していること
```

### 5. セキュリティ上の問題があるコードが生成される

**症状**: SQLインジェクション、XSS、認証の欠落などの脆弱性

**対処法**:
- CLAUDE.mdにセキュリティ要件を明記
- PR前のAIセルフレビューでセキュリティ観点を必ず含める
```
# CLAUDE.mdに追記
## セキュリティ要件
- ユーザー入力は必ずバリデーション
- SQLはPrismaのORMを使用（raw SQL禁止）
- 認証が必要なエンドポイントには必ず認証ミドルウェアを適用
```

## 今週から始めること

### Day 1：CLAUDE.mdを作る

```bash
touch CLAUDE.md
git add CLAUDE.md
git commit -m "docs: add CLAUDE.md for AI-driven development context"
```

チームのContributing.mdやNotion等に以下を追記：

```markdown
## PRを出す前に
git diff main | claude "セキュリティ・パフォーマンス・規約違反の観点でレビューして"
を実行し、指摘事項を修正してからPRを出すこと。
```

## コスト管理のヒント

AI駆動開発では API 利用料金が発生します。効率的に使うためのヒント:

- **コンテキストを絞る**: 必要なファイルだけを参照させる
- **小さく始める**: 大きなタスクは分割して依頼
- **CLAUDE.mdを充実させる**: 毎回説明する手間とトークンを削減
- **キャッシュを活用**: 同じセッション内で関連タスクをまとめて実行
