## AI駆動開発とは？

### 一言で言うと

「AIが開発者のパートナーとなり、コードを書く・レビューする・設計を相談する作業を支援する開発スタイル」です。

### 従来の開発との違い

| 従来の開発 | AI駆動開発 |
|-----------|-----------|
| エンジニアが1行ずつコードを書く | AIが下書きを作成し、エンジニアが確認・修正 |
| ドキュメントは後回しになりがち | AIがドキュメントを自動生成 |
| 設計の相談相手が限られる | 24時間いつでもAIに設計相談が可能 |
| コードレビューは人間同士のみ | AIが事前レビューで問題を発見 |

### AIは何ができる？

1. **コードの生成**: 「ログイン機能を作って」→ 動くコードを生成
2. **コードの説明**: 既存コードの動作を分かりやすく解説
3. **バグの発見**: エラーログを渡すと原因と解決策を提案
4. **設計の相談**: 「この機能はどう実装すべき？」に選択肢と根拠を提示
5. **ドキュメント作成**: コードからREADMEやAPI仕様書を自動生成

## なぜ今AI駆動開発なのか？

### 1. 開発効率が向上する

- 定型的なコードはAIが数秒で生成
- ドキュメント作成の時間が大幅に短縮
- バグ調査の初動がスピードアップ

### 2. 品質が安定する

- AIがコードレビューで見落としを減らす
- コーディング規約の遵守を自動チェック
- セキュリティ上の問題を早期発見

### 3. 知識格差が縮まる

- 新人でもAIのサポートで難しいタスクに挑戦できる
- 技術的な質問への回答が即座に得られる
- 属人化していた知識がドキュメント化される

## チームに導入すると何が変わるのか

### Before: AI導入前

```
PM: 「この機能どのくらいで実装できる？」
エンジニア: 「調査が必要なので、見積もりに1日ください」
（1日後）
エンジニア: 「技術的に3パターンあって...」
PM: 「どれがいいの？」
エンジニア: 「メリット・デメリットを整理するのにもう少し...」
```

### After: AI導入後

```
PM: 「この機能どのくらいで実装できる？」
エンジニア: 「AIと壁打ちしてきます」
（30分後）
エンジニア: 「3パターンの比較表とおすすめ案をまとめました」
PM: 「早い！じゃあこれで進めよう」
```

### 具体的な変化

| 場面 | 変化 |
|------|------|
| 技術調査 | 数日→数時間に短縮 |
| 設計相談 | 人を探す手間→即座にAIと壁打ち |
| コードレビュー | 人の負担軽減→AIが一次チェック |
| ドキュメント作成 | 後回し→実装と同時に自動生成 |
| 新人教育 | 先輩の時間を占有→AIが補助 |

## 主要なツール

### Cursor（カーソル）

- **何か**: AIが組み込まれたコードエディタ
- **イメージ**: WordにAIアシスタントが常駐しているような感覚
- **特徴**: 画面上でコードを見ながらAIと対話できる

### Claude Code（クロード・コード）

- **何か**: ターミナル（黒い画面）で動くAIエージェント
- **イメージ**: コマンドで指示を出すと、AIが自律的にファイルを編集
- **特徴**: 大規模な変更や自動化に強い

### 使い分け

| ツール | 適した場面 |
|--------|-----------|
| Cursor | 画面を見ながら対話的に作業したい時 |
| Claude Code | 大量のファイルを一括で変更したい時 |

## 非エンジニアとして知っておくべきポイント

### 1. AIは「万能」ではない

- **できること**: パターン化された作業、定型的なコード生成、既存情報の整理
- **苦手なこと**: ビジネス要件の判断、創造的な意思決定、プロジェクト固有の文脈理解

**ポイント**: AIはツールであり、最終判断は人間が行います。

### 2. 「CLAUDE.md」がチームの共有脳になる

プロジェクトのルールや技術スタックを書いたファイル（CLAUDE.md）をリポジトリに置くと、AIが自動的に読み込みます。

**ポイント**: このファイルにビジネスの文脈や制約を書いておくと、AIの回答精度が上がります。PMやデザイナーの方も「こういう情報を追加してほしい」とエンジニアに伝えてください。

### 3. セキュリティへの配慮が必要

- APIキーやパスワードをAIに渡す際は注意
- 機密情報がコードに含まれていないかチェック

**ポイント**: エンジニアが「.envファイルに書く」「.gitignoreに追加する」と言ったら、セキュリティ対策をしているということです。

### 4. AIへの指示の出し方が品質を左右する

曖昧な指示 → 曖昧な結果
具体的な指示 → 具体的な結果

**ポイント**: エンジニアが「要件をもう少し詳しく」と聞いてきたら、AIへの指示精度を上げるためです。

## PMとして活用できる場面

### 1. 技術的な質問への素早い回答を得る

エンジニアに聞く前に、AIに「この機能は技術的に実現可能？」と聞いてみる。完璧な回答でなくても、質問の解像度が上がります。

### 2. 仕様のドラフト作成

「こういう機能の仕様書をドラフトして」とAIに依頼し、叩き台を作成。エンジニアとの議論が効率化します。

### 3. 進捗報告資料の作成支援

「このコミットログを元に、今週の進捗をまとめて」とAIに依頼。

---

## デザイナーとして活用できる場面

### 1. 実装可能性の確認

「このデザインはReactで実装可能？難しい点はある？」とAIに確認。

### 2. CSSの生成

デザインカンプから「このレイアウトのCSSを書いて」と依頼。

### 3. デザインシステムとの整合性チェック

「このコンポーネントは既存のデザインシステムと一貫性がある？」と確認。

## 経営者・マネージャーとして知っておくべきこと

### コストについて

- AIツールは従量課金制が多い（使った分だけ課金）
- 月額固定のプランもある
- 開発効率向上とのトレードオフで判断

### リスク管理

- 機密情報の取り扱いルールを策定する
- AIの出力を盲信せず、人間のレビューを必須にする
- 著作権やライセンスへの配慮

### 導入効果の測定

以下の指標で効果を測定できます：

- プルリクエストのリードタイム
- バグの発生率
- ドキュメントのカバレッジ
- エンジニアの満足度調査

## よくある質問

### Q: エンジニアの仕事がなくなるの？

いいえ。AIは「ツール」であり、エンジニアの判断力・創造力・ビジネス理解は引き続き必要です。むしろ、単純作業から解放され、より高度な仕事に集中できるようになります。

### Q: 導入コストはどのくらい？

ツール利用料は月額数千円〜数万円程度。ただし、導入初期は学習コストがかかります。効果が出始めるまで1〜2ヶ月程度を見込んでください。

### Q: セキュリティは大丈夫？

適切な運用ルールを設ければ問題ありません。機密情報の取り扱い、コードの外部送信に関するポリシーをエンジニアと一緒に策定してください。

### Q: どのプロジェクトから始めるべき？

新規プロジェクトか、技術的負債が少ない小規模プロジェクトがおすすめです。既存の大規模プロジェクトへの導入は慎重に。
